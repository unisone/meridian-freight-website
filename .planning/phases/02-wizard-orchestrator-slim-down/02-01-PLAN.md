<deviation_notes>
PATTERNS.md proposes two separate files `wizard/step-contact.tsx` and `wizard/step-result.tsx`.
This plan intentionally MERGES them into a single file `wizard/estimate-card.tsx` (Task 5a).
Rationale (carried forward from `01-01-PLAN.md`):

1. The existing `CalculatorV3EstimateCard` (current orchestrator lines 1003–~1620) already
   combines the email-gate (contact) form and the estimate-result display in one cohesive UI
   element. The two surfaces share local state and conditionally swap inside the same card.
2. Splitting them into two files would break the atomicity of the email-gate UX state machine
   (lead-form → submit → estimate-display lives as one component lifecycle, not two).
3. The 500-line cap still applies. If `estimate-card.tsx` exceeds 500 lines during execution,
   the sanctioned escape hatch is to split presentational helpers into a sibling
   `wizard/estimate-card-helpers.tsx` (no JSX-tied state, just sub-components like `DetailRow`,
   `SectionHeader`, `ImportCostNote`) — NOT to split contact/result back apart.
4. Pattern compliance is preserved on every other file (`step-equipment.tsx`, `step-specs.tsx`,
   `step-route.tsx`).
