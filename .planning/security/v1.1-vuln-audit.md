# v1.1 Phase 4 — Dependabot Vulnerability Audit

**Date:** 2026-05-04
**Phase:** 04
**Goal:** Resolve 2 moderate Dependabot advisories on `main` (PR #107 push surfaced 2 vulns; later expanded to 9 transitive cascades on `npm audit`).
**Outcome:** 1 advisory patched, 1 documented as residual with explicit rationale.

## Pre-State (after PR #110 merged)

```
npm audit --audit-level=moderate
9 moderate severity vulnerabilities (cascading from 2 root causes)
```

| Root advisory | GHSA | Affected ranges | Cascading deps |
|---|---|---|---|
| postcss XSS via `</style>` in CSS Stringify | GHSA-7fh5-64p2-3v2j | `<8.4.31` | `next`, `next-intl`, `@sentry/nextjs`, `@vercel/analytics`, `@vercel/speed-insights` |
| uuid: bounds check in v3/v5/v6 when buf provided | GHSA-mc76-5925-c5p6 | `<14.0.0` (advisory range; no version >=14 exists) | `svix` (used internally by `resend`) |

## Resolution

### postcss — PATCHED (Dependabot #77)

Added `package.json` overrides:

```json
"overrides": {
  "postcss": "^8.5.0"
}
```

Verified: every install of `postcss` in the lockfile is now at `8.5.14` (≥8.4.31, includes the XSS fix). 0 remaining advisories for `postcss`.

### uuid — RESIDUAL (documented, accepted)

Added override:

```json
"overrides": {
  "uuid": "^11.0.0"
}
```

Result: all `uuid` instances now at `11.1.1` (latest stable). However, the GHSA-mc76-5925-c5p6 advisory's affected-version range is `<14.0.0` — and **no `uuid` v12/13/14 exists**. The advisory database is being defensive about an upstream issue that doesn't have a forward fix yet. `npm audit` will keep flagging this regardless of which version of `uuid` we install.

**Why we accept the residual risk:**

1. **`uuid` is a transitive dependency, not a direct one.** We do not import `uuid` anywhere in `app/`, `components/`, or `lib/`. Verified:
   ```bash
   grep -rn 'from .uuid\|require..uuid' app/ components/ lib/ 2>&1
   # → zero matches
   ```

2. **`svix` is the only consumer in our tree** — it's the SDK that `resend` uses internally for webhook signature verification. We use `resend` strictly for transactional email (contact form notifications + auto-replies + calculator submissions). Verified:
   ```bash
   grep -rn 'svix\|webhook' app/ components/ lib/ 2>&1
   # → zero matches in our code
   ```

3. **The vulnerable code path is unreachable from us.** The advisory says: "Missing buffer bounds check in v3/v5/v6 when `buf` is provided." We:
   - Do not call `uuid` directly anywhere
   - Do not pass a `buf` argument anywhere (the API surface that triggers the vuln)
   - Use `resend.emails.send()` only — the email API path, which doesn't invoke webhook signing

4. **The advisory range is functionally unbounded.** If there's no `uuid@14`, "fix this advisory" is not actionable through dependency upgrades. Pinning to the latest stable (`11.1.1`) is the most we can do without ripping out `resend`.

**Mitigation strategy if exposure changes:**

- If we ever add webhook handling (e.g. for incoming Resend events, or any direct `svix` usage), re-audit immediately. Replace `resend` with a direct HTTP integration that doesn't pull `svix`.
- Subscribe to `svix` and `uuid` upstream advisories. When `uuid@14` ships (or when `svix` decides to switch to `crypto.randomUUID()`), upgrade.
