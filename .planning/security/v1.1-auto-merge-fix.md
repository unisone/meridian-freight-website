# Auto-merge Dependabot — Lint Gate Fix

**Date:** 2026-05-05
**Trigger:** Dependabot PR #105 (eslint 9 → 10 bump) merged to `main` on 2026-04-27 with **FAILING** `Lint, Build & Test`. Result: `main` was red for ~7 days; every downstream PR's CI inherited the broken lint.

## Root Cause

Three contributing factors, layered:

### 1. Branch protection on `main` required ZERO status checks (the platform hole)

Pre-fix state (queried via `gh api repos/.../branches/main/protection`):
```json
{
  "required_status_checks": { "strict": false, "contexts": [], "checks": [] },
  "required_pull_request_reviews": { "required_approving_review_count": 0 },
  "enforce_admins": { "enabled": false }
}
```

Result: `gh pr merge --auto --squash` had no gate to wait on. GitHub merged immediately because no required check was missing.

### 2. wait-on-check-action@v1.6.1 had a race condition

The auto-merge workflow correctly used `lewagon/wait-on-check-action` with `allowed-conclusions: success`. But v1.6.1 has a known race where, if the named check hasn't *started* yet at the time the action runs, the action exits with success instead of waiting for the check to start. Combined with branch protection requiring nothing, the merge proceeded.

### 3. The merge step was a single command (no defense in depth)

`gh pr merge --auto --squash "$PR_URL"` — fire and forget. No second sanity check between "the wait action returned" and "the merge fires."

## Fix (this PR)

### Layer 1 — Branch protection (the durable platform fix)

`PUT repos/.../branches/main/protection` with:
```json
{
  "required_status_checks": { "strict": false, "contexts": ["Lint, Build & Test"] },
  "enforce_admins": false,
  "required_pull_request_reviews": null,
  "restrictions": null,
  "allow_force_pushes": false,
  "allow_deletions": false
}
```

`Lint, Build & Test` is now a **required** status check on `main`. GitHub will refuse to merge any PR (including `--auto` Dependabot merges) until that check passes. This is the most important fix and the one that closes the actual hole.

Verified post-update:
```
{
  "allow_force_pushes": false,
  "enforce_admins": false,
  "required_contexts": ["Lint, Build & Test"]
}
```

### Layer 2 — Bump wait-on-check-action to v1.7.0

v1.7.0 release notes specifically address race conditions and adds `verbose` output for diagnostics. This matches the open Dependabot PR #89 (which is now superseded by this PR).

### Layer 3 — Explicit pre-merge re-verification

Added a step that uses `gh api` to re-read the check-runs for the PR's HEAD SHA and asserts that `Lint, Build & Test` conclusion is exactly `"success"`. If anything raced between the wait action and the merge call, this step fails before `gh pr merge` runs.

## Why three layers

Branch protection alone closes the hole. But:
- Branch protection is invisible to the workflow itself (no log line shows it caught a regression). Layers 2 + 3 give the workflow run a clear failure signal in CI logs if the gate ever has to engage.
- Layer 3 also catches misconfigurations: if someone ever renames the workflow check or removes it from required-status-checks, the workflow fails loudly instead of silently letting bad merges through.

## Verification (operational)

- [x] `gh api .../branches/main/protection` shows `Lint, Build & Test` in `required_status_checks.contexts`
- [x] Workflow YAML uses `lewagon/wait-on-check-action@v1.7.0`
- [x] Workflow has explicit `gh api` re-verification step before `gh pr merge`
- [x] Permissions added: `checks: read` for the new gh api call

## What This Doesn't Fix (deferred)

- **Required PR review approvals** — currently `required_approving_review_count: 0`. Acceptable for a solo-developer project with the autonomy directive. Would re-evaluate on team scaling.
- **Linear history requirement** — currently disabled. Squash-merge already enforces a clean history.
- **Other workflows** (`Lighthouse CI`, `Performance Audit`) — currently informational, not required. Keeping them informational is correct because they can be flaky and aren't critical-path gates.
